# せどりAIシステム MVP実装方針

## MVPの目的

このMVPの目的は、Amazon SP-APIの審査・接続完了を待たずに、せどり実務で最も価値が出る「利益候補の発見、判断、仕入れ記録、在庫化」までの業務ループを検証することです。

初期ユーザーは、日本国内のせどり初心者から中級者、小規模物販事業者を想定します。MVPでは、完全自動化よりも「判断材料を早く揃える」「仕入れミスを減らす」「作業履歴を残す」ことを優先します。

成功条件は次の通りです。

- ユーザーが商品URL、JAN、ASIN、CSV、手入力のいずれかから商品候補を登録できる。
- Amazon、メルカリ、ヤフオク、楽天、Yahoo等の価格情報を同一画面で比較できる。
- 利益額、ROI、送料、手数料、回転リスク、出品者増加、Amazon本体在庫などを加味して、AIまたはルールでA/B/C/NG判定を出せる。
- 有望候補を仕入れ予定、仕入れ実績、在庫に変換できる。
- 外部APIが未接続でも、手動入力・CSV取込・外部リンク保存で主要画面が動く。
- 後からAmazon SP-API、Keepa、会計API、通知APIを追加しても、商品・価格・候補・在庫のデータ構造を大きく変更しない。

## MVP方針の前提

- MVPは「公式API連携ありき」ではなく、「手動・CSV・一部APIで成立する業務アプリ」として作る。
- Amazon SP-APIは重要だが、取得難易度と審査待ちのリスクがあるため、Phase 1の必須条件にしない。
- 価格取得や相場確認は、初期は以下を組み合わせる。
  - ユーザーによるURL入力
  - JAN/ASIN/商品名の手入力
  - CSVインポート
  - Keepa API
  - 楽天/Yahooなど取得しやすいAPI
  - 管理者またはユーザーによる価格スナップショット登録
- メルカリ、ヤフオクはMVPでは自動購入・自動出品・自動操作を行わない。URL保存、手動補完、CSV、外部画面への遷移を基本にする。
- AI判定は、最初から高度な学習モデルを作らず、ルールベース計算とLLMコメントを組み合わせる。
- 監査ログ、APIキー保護、利用量記録は最小限から入れる。外部キーや課金に関わるため、後付けにしすぎない。

## Phase 1〜4の実装順序

### Phase 1: 価格差発見MVP

目的は、ユーザーが商品候補を登録し、横断比較し、仕入れるか見送るかを判断できる状態を作ることです。

実装対象:

- ログイン/組織選択の最小実装
- API連携設定の土台
  - APIキー登録
  - 接続状態表示
  - 検証結果の保存
  - シークレットを画面に再表示しない設計
- 商品マスタ作成
  - 商品名
  - JAN/ASIN/型番/SKU
  - 画像URL
  - 商品状態
  - メモ
- 価格スナップショット登録
  - チャネル
  - URL
  - 価格
  - 送料
  - ポイント
  - 在庫有無
  - セラー数
  - Amazon本体在庫フラグ
  - 取得日時
- 横断比較画面
  - 商品単位の販路別価格比較
  - 仕入れ候補チャネルと販売候補チャネルの比較
  - 手数料、送料、想定利益、ROIの表示
- 価格差チャンス一覧
  - 利益額順
  - ROI順
  - 判定順
  - リスクありフィルタ
- AI/ルール判定
  - A: 仕入れ候補
  - B: 要確認
  - C: 見送り寄り
  - NG: 仕入れ不可
  - 判定理由の短文表示
- CSVインポート/エクスポートの最小実装
  - 商品候補取込
  - 価格スナップショット取込
  - チャンス一覧出力

Phase 1では、在庫・注文・会計は本格実装しません。ただし、後続Phaseにつなげるため、候補のステータスは `new`, `watching`, `approved`, `rejected` 程度まで持たせます。

### Phase 2: 仕入れ・在庫MVP

目的は、利益候補を実際の仕入れ予定/実績に変換し、在庫として追跡できる状態を作ることです。

実装対象:

- 仕入れ登録
  - 仕入先
  - 仕入価格
  - 数量
  - 送料
  - 仕入日
  - 証憑URL/メモ
  - 仕入れ候補からの変換
- 入庫処理
  - 仕入れ実績から在庫アイテム生成
  - SKU採番
  - 状態、保管場所、原価の登録
- 在庫一覧/在庫詳細
  - 在庫ステータス
  - 原価
  - 滞留日数
  - 出品予定価格
  - 在庫移動履歴
- 手動の販売/注文登録
  - 販売チャネル
  - 販売価格
  - 手数料
  - 送料
  - 発送状態
- ダッシュボード
  - 価格差候補数
  - 承認済み仕入れ候補
  - 在庫数
  - 滞留在庫
  - エラー/未処理件数

Phase 2では、Amazonや会計ソフトへの自動連携はまだ必須にしません。手入力とCSVで「業務の型」が回ることを確認します。

### Phase 3: 自動取得・ジョブ・通知

目的は、手作業で成立した業務ループに自動取得と定期実行を追加し、日次運用に耐える状態にすることです。

実装対象:

- ジョブ基盤
  - 手動実行
  - 定期実行
  - 実行履歴
  - 失敗理由
  - 再実行
- Keepa API連携
  - ASINベースの価格/ランキング取得
  - Amazon本体在庫やセラー数の取得
  - 利用量制限の記録
- 楽天/Yahoo等の検索API連携
  - JAN/キーワード検索
  - 価格候補の取込
- アラート
  - 利益率低下
  - 価格急落
  - 滞留在庫
  - API接続エラー
- 通知
  - まずはアプリ内通知
  - 次にメールまたはSlack/LINEのどれか1つ
- AI判定の改善
  - 過去の承認/却下フィードバック保存
  - 判定理由のテンプレート化
  - NG条件の管理画面化

Phase 3で初めて、外部APIによる自動取得の比率を上げます。ただし、外部APIが失敗しても手動価格登録に戻れる設計を維持します。

### Phase 4: 公式連携・会計・課金

目的は、MVPで検証できた業務価値を、公式API連携・会計出力・課金管理まで拡張することです。

実装対象:

- Amazon SP-API連携
  - 注文インポート
  - 出品情報取得
  - 在庫/価格同期
  - フィードバックやエラー状態の保存
- 会計CSV/レポート
  - 売上
  - 原価
  - 手数料
  - 送料
  - 返品
  - 月次集計
  - freee/Money Forward向けCSV
- セキュリティ/監査の強化
  - APIキーアクセス履歴
  - 監査ログ検索
  - 権限変更履歴
  - 重要操作の確認
- 利用量/課金
  - API利用量
  - AI利用量
  - プラン別上限
  - 請求状態
- 出品・注文同期の拡張
  - Amazon以外の販路連携
  - CSV取込テンプレートの追加
  - エラーリカバリ導線

Phase 4は、事業として継続課金する段階で必要になる機能を中心にします。MVP価値検証前に作り込みすぎないことが重要です。

## Amazon SP-APIなしでも動くMVP方針

Amazon SP-APIが未取得でも、以下の代替導線でMVPを成立させます。

| 領域 | SP-APIありで行うこと | SP-APIなしのMVP代替 |
|---|---|---|
| 商品特定 | ASIN、SKU、出品情報から自動取得 | ASIN/JAN/商品名/URLを手入力、CSV取込 |
| Amazon価格 | 公式APIで価格や出品状態を取得 | Keepa API、手入力、CSV、外部確認リンク |
| Amazon本体在庫 | 公式情報を参照 | Keepaまたはユーザー入力のフラグ |
| セラー数/ランキング | APIから取得 | Keepa APIを優先。なければ任意入力 |
| 注文取込 | SP-APIで注文同期 | 手動登録、CSVインポート |
| 在庫同期 | Amazon在庫と同期 | 内部在庫を正とし、出品状況は手動更新 |
| 出品作成 | APIで出品/更新 | 出品予定情報を管理し、外部画面へ遷移 |
| 会計連携 | 注文・手数料を自動反映 | 手動注文、CSV、月次集計で代替 |

設計上のポイント:

- `market_prices` は「API取得結果」だけでなく「手入力価格」も保存できる前提にする。
- `source_channel`, `source_url`, `raw_payload`, `fetched_at` を使い、取得元がAPIでも手動でも同じ画面で扱えるようにする。
- `channel_product_matches` は、AI推定やJAN一致だけでなく、ユーザー確認済みの手動マッチを重視する。
- `cross_channel_opportunities` は、公式APIの有無に関係なく、価格スナップショットから生成できるようにする。
- 自動取得できない項目は空欄にせず、「未取得」「手動確認」「要再確認」などの状態を明示する。
- 画面上では「API未接続のため一部手動入力」と分かる表示にし、ユーザーが不具合と誤解しないようにする。

## 最初に実装する機能

最初に実装すべき機能は、価格差発見から仕入れ判断までの最短導線です。

優先度1:

- ログイン/組織選択の簡易版
- 商品候補の手入力登録
- JAN/ASIN/URL/商品名の保存
- 価格スナップショットの手入力登録
- CSVインポート
- 横断比較テーブル
- 利益額/ROI/送料/手数料の計算
- A/B/C/NGの初期判定
- 判定理由の表示
- 価格差チャンス一覧
- 候補の承認/却下

優先度2:

- APIキー登録画面
- Keepa API接続確認
- 取得ジョブの手動実行
- 商品詳細画面
- 価格推移の簡易表示
- ダッシュボードのKPI
- 仕入れ予定への変換

優先度3:

- 在庫化
- 注文手動登録
- ジョブ履歴
- アプリ内アラート
- 会計CSVの簡易出力

## 後回しにする機能

以下はMVP初期では後回しにします。

- Amazon SP-APIの完全連携
- 自動出品
- 自動価格改定
- 自動購入
- メルカリ/ヤフオクの自動操作
- 複雑な在庫ロケーション管理
- 返品/返金の詳細ワークフロー
- freee/Money ForwardのAPI直接連携
- 高度な課金プラン管理
- チーム招待、権限変更、監査ログの詳細画面
- AIの個別学習やファインチューニング
- 画像類似検索
- バーコードスキャンアプリ
- モバイル専用UI
- 多通貨/海外販路対応

後回しにする理由は、MVPの価値検証に直結しないか、外部審査・規約・運用リスクが大きいからです。特に自動購入や自動出品は、誤操作時の損失が大きいため、手動確認を挟む設計が固まってから検討します。

## リスクと回避策

| リスク | 影響 | 回避策 |
|---|---|---|
| Amazon SP-APIの審査が遅れる | 注文/出品/在庫同期が使えない | SP-APIなしで成立する手動・CSV・Keepa中心のMVPにする |
| Keepa APIの利用量制限に到達する | 価格/ランキング取得が止まる | 利用量ログ、キャッシュ、手動再取得、優先度付きジョブを入れる |
| 価格情報が古くなる | 誤仕入れにつながる | `fetched_at` を明示し、古い価格は警告する |
| 手数料・送料計算が不正確 | 利益額がズレる | 初期は保守的な係数を使い、ユーザーが上書きできるようにする |
| 商品マッチングの誤判定 | 別商品を比較してしまう | JAN/ASIN一致を最優先し、タイトル類似は要確認にする |
| AI判定を過信される | 損失やクレームにつながる | 判定理由、未取得項目、リスク警告、最終判断はユーザー確認にする |
| 外部サイト規約に抵触する | サービス停止や法務リスク | 自動スクレイピング/自動操作を避け、公式API・手動入力・CSVを優先する |
| APIキー漏えい | ユーザー被害、信用低下 | 暗号化保存、再表示禁止、アクセスログ、ローテーション導線を入れる |
| MVP範囲が膨らむ | リリースが遅れる | Phaseごとの完了条件を固定し、Phase 1は価格差発見に集中する |
| 会計・税務要件が複雑化する | 実装負荷が増える | 初期はCSVと月次集計に留め、税務判断は専門家確認前提にする |

## APIキー取得の優先順位

APIキーや外部連携は、MVP価値への寄与と取得難易度のバランスで優先順位を付けます。

### 優先度A: すぐ必要

1. AI APIキー
   - 用途: 判定理由生成、リスクコメント、候補要約。
   - 理由: MVPの差別化要素になりやすい。
   - 代替: ルールベース判定のみでも動くようにする。

2. Keepa APIキー
   - 用途: Amazon価格、ランキング、セラー数、Amazon本体在庫の補助取得。
   - 理由: SP-APIなしMVPでAmazon相場を見るための中心になる。
   - 代替: 手入力、CSV、外部リンク確認。

3. メール送信APIまたは簡易通知基盤
   - 用途: 招待、アラート、ジョブ失敗通知。
   - 理由: 運用エラーを見逃さないため。
   - 代替: アプリ内通知のみ。

### 優先度B: Phase 2〜3で必要

4. 楽天商品検索系API
   - 用途: JAN/キーワードによる楽天価格候補取得。
   - 理由: 取得しやすく、横断比較の厚みを出せる。
   - 代替: URL入力、CSV。

5. Yahooショッピング系API
   - 用途: Yahoo側の価格候補取得。
   - 理由: 国内比較対象として重要。
   - 代替: URL入力、CSV。

6. Slack/LINE等の通知API
   - 用途: 価格変動、仕入れ候補、ジョブ失敗通知。
   - 理由: 日次運用に入ると価値が出る。
   - 代替: アプリ内通知、メール。

### 優先度C: Phase 4で取得

7. Amazon SP-API
   - 用途: 注文、出品、在庫、価格、レポート連携。
   - 理由: 本格運用では重要だが、審査と実装負荷が高い。
   - 代替: Keepa、CSV、手動登録。

8. freee/Money Forward API
   - 用途: 会計ソフトへの直接連携。
   - 理由: MVPではCSVで代替できる。
   - 代替: 会計CSVエクスポート。

9. 決済/課金API
   - 用途: 有料プラン、請求、利用量課金。
   - 理由: 継続課金開始前に必要。
   - 代替: 手動請求、無料トライアル運用。

## 2週間ロードマップ

2週間のゴールは、SP-APIなしで価格差候補を登録し、比較し、仕入れ判断できるプロトタイプを完成させることです。

Week 1:

- 認証/組織の最小実装
- 商品マスタ、商品識別子、価格スナップショットのCRUD
- 商品候補の手入力フォーム
- 価格スナップショットの手入力フォーム
- CSVインポートの最小実装
- 横断比較の利益計算ロジック
- 手数料/送料/ROIの初期計算ルール

Week 2:

- 横断比較画面
- 価格差チャンス一覧
- A/B/C/NG判定ロジック
- AIコメント生成またはルールベースコメント
- 候補の承認/却下
- 商品詳細画面の最小版
- APIキー登録画面の土台
- サンプルデータでの業務フロー確認

2週間時点の完了条件:

- 手入力またはCSVで10〜50件の商品候補を登録できる。
- 価格差候補が一覧化され、利益額/ROI順で並べられる。
- ユーザーが候補ごとに「仕入れる/見送る」を判断できる。
- SP-APIがなくてもデモが成立する。

## 4週間ロードマップ

4週間のゴールは、仕入れ予定から在庫化までつなげ、少人数の実運用テストに耐える状態にすることです。

Week 3:

- Keepa API接続確認
- Keepa取得ジョブの手動実行
- 取得結果を `market_prices` に保存
- 価格取得日時と古い価格の警告
- ダッシュボードの最小KPI
- API接続エラー表示

Week 4:

- 仕入れ候補から仕入れ予定への変換
- 仕入れ登録
- 入庫処理
- 在庫一覧/詳細
- 在庫ステータス更新
- 注文の手動登録
- ジョブ実行履歴の最小表示

4週間時点の完了条件:

- 価格差発見から仕入れ登録、在庫化まで一連の操作ができる。
- Keepa APIが接続済みならAmazon相場情報を半自動で補完できる。
- APIが失敗しても手動入力で業務が止まらない。
- 1〜3名程度のテストユーザーが日次で使える。

## 8週間ロードマップ

8週間のゴールは、実運用の継続利用に必要な自動化、通知、会計出力、セキュリティを整え、有料化判断に進める状態にすることです。

Week 5〜6:

- 定期ジョブ
- ジョブ失敗時の再実行
- アプリ内アラート
- 利益低下/価格急落/滞留在庫アラート
- 楽天/Yahoo API連携の追加
- AI判定フィードバック保存
- CSVテンプレートの整備

Week 7〜8:

- 会計CSVの簡易出力
- 月次集計
- 利用量ログ
- APIキーアクセスログ
- 監査ログの最小表示
- Amazon SP-API申請/検証環境の準備
- 課金プランの仮設計
- βユーザー向け運用手順の作成

8週間時点の完了条件:

- テストユーザーが毎日候補確認、仕入れ判断、在庫確認を行える。
- 価格取得、AI判定、通知の一部が自動化されている。
- 会計CSVまたは月次集計を出せる。
- API利用量と主要操作ログを確認できる。
- Amazon SP-API連携を追加するためのデータ構造と画面導線が準備できている。

## 実装判断の基準

MVPでは、機能追加の判断を次の基準で行います。

- 価格差発見の精度が上がるか。
- 仕入れミスを減らせるか。
- ユーザーの手入力を減らせるか。
- APIなしでも代替操作があるか。
- 後から公式APIに差し替えられるか。
- 誤操作時の金銭的損失を防げるか。
- 2週間以内に検証できる粒度か。

この基準に合わない機能は、Phase 4以降に回します。最初のMVPでは「全部つながっているが薄い」よりも、「価格差発見と仕入れ判断だけは実務で使える」状態を優先します。
